## 1. 总体评价

系统整体实现规范、结构清晰，核心积分逻辑（双打/单打独立、零和、赛制权重、比分系数）与设计稿一致，且已修复前一轮评审指出的若干问题（写入原子性、胜方推导、注入防护等）。

主要剩余风险集中在**两个方面**：

1. **写入模型**采用「整库快照覆盖」，在多人并发录入时会发生数据丢失——这是本次审计唯一的高危项。
2. **数据质量与安全细节**：比分缺少按赛制的合法性校验、规则快照未生效、登录缺少加固、缺少操作审计。

整改成本普遍较低，建议优先处理高危项与比分校验。

---

## 2. 系统概况

- 架构：单进程 Node.js（原生 `http`）+ `better-sqlite3`，前端为手机优先单页 HTML。
- 鉴权：单一管理口令 `TENNIS_ADMIN_PASSWORD`，登录后发放内存会话 Cookie；读公开、写需鉴权。
- 数据：SQLite 四表 `players` / `matches` / `player_ratings` / `rating_history`；后两者为派生缓存，真值是 `players` + `matches`。
- 写入：前端持有全量数据，任何增删都把整份 `{players, matches}` 经 `PUT /api/state` 上传，后端清空全表后重灌并全量重算。
- 部署：Docker / docker-compose，数据卷挂载 `/data`，内置在线备份脚本。

## 3. 已确认的良好实践

| # | 项 | 说明 |
|---|---|---|
| G1 | 写入原子事务 | `rewriteState` 用 `db.transaction` 包裹，清空+重灌+重算要么全成功要么全回滚 |
| G2 | SQLite 健壮配置 | 开启 `foreign_keys`、`journal_mode=WAL`、`busy_timeout=5000` |
| G3 | 安全启动约束 | 未设置 `TENNIS_ADMIN_PASSWORD` 时进程直接退出，避免裸奔 |
| G4 | 胜方由比分推导 | `winner_team = sa > sb ? 'A' : 'B'`，不接受手填，消除脏数据风险 |
| G5 | 注入防护 | 首屏数据内联前经 `safeJsonForHtml` 转义，防止 `</script>` 注入 |
| G6 | CSRF 基本缓解 | 会话 Cookie 使用 `HttpOnly` + `SameSite=Lax`，挡住大部分跨站写请求 |
| G7 | 备份与可恢复性 | 内置在线备份脚本；因缓存全派生，仅备份 `players`+`matches` 即可完整恢复 |
| G8 | 索引合理 | `matches(mode,date,id)`、`rating_history(player_id,mode)` 覆盖主要查询路径 |
| G9 | 积分逻辑正确 | 零和恒等成立（单打 1+1、双打 2+2），删除经外键级联 + 全量重算保持路径一致 |

### H-1 🔴 整库覆盖写入导致并发丢数据

**位置**：`server.js` `rewriteState`（约 348–463 行，372–375 行 `DELETE` 全表）；`tennis_ranks.html` `persistState`（约 505–511 行）。

**描述**：前端在内存中持有全量 `PLAYERS` / `MATCHES`，任何新增或删除都把整份数据经 `PUT /api/state` 上传，后端先 `DELETE` 四表再重灌。写入是「整库最后写入者覆盖」，且无版本号 / ETag 等并发控制。

**影响**：两名管理员各自打开页面后，A 录入一场并保存，B（持有 A 保存前的旧数据）随后保存，B 的整份快照会覆盖数据库，**A 的那场比赛被静默丢失**。单人记分无影响，多人在手机端各自录入时极易触发，且无任何报错。

**建议**：二选一。
1. 改为增量接口：`POST /api/matches`（写单场后重算）、`DELETE /api/matches/:id`（删单场后重算）、`POST /api/players` 等，前端不再上传全量。
2. 保留全量 PUT，但引入乐观锁：`/api/state` 返回一个版本号（如最大 `updated_at` 或自增版本），PUT 时回带；版本不一致则拒绝（409）并提示前端刷新后重试。

> 推荐方案 1，顺带解决 L-1。

### M-1 🟠 比分未按赛制做合法性校验

**位置**：`server.js` `rewriteState` 比赛规范化段（约 396–427 行）。

**描述**：当前仅校验「比分为整数、A≠B」，未校验比分是否符合所选赛制。`标准一盘 70:4`、`抢7 打成 2:1`、`抢11 5:3` 等非法比分都能通过，`scoreMultiplier` 还会按兜底分支给出一个系数。

**影响**：错误输入被静默接受并参与积分计算，污染核心结果（积分），且事后难以发现。

**建议**：增加按赛制的合法比分校验，不合法直接拒绝。规则：
- 抢 7：胜方 = 7 且净胜 ≥2，或更高且净胜 2；
- 抢 11：胜方 = 11 且净胜 ≥2，或更高且净胜 2；
- 四局短盘：胜方 ∈ {4,5}，对应 4:0~4:3 / 5:3 / 5:4；
- 标准一盘：6:0~6:4 / 7:5 / 7:6。

---

### M-2 🟠 规则快照列未被使用，改规则会回溯重写历史

**位置**：`server.js` `computeMatchDeltas`（219 行现算系数）与快照写入（424–425 行）。

**描述**：`matches` 表保存了 `format_weight` / `score_multiplier` 快照，但全量重算时 `computeMatchDeltas` 是用当前代码**现算**系数，从不读取快照；且每次整库重写又用当前规则覆盖快照。结果：调整一次系数表，所有历史比赛会被按新规则回溯重算，快照永远等于「当前规则」，无法冻结历史。

**影响**：快照列形同虚设，且「改规则即改写全部历史积分」这一行为未被明示，易在调参时意外推翻既往战绩。

**建议**：明确并保持一致，二选一。
1. 冻结历史：重算时改为读取每场快照的 `score_multiplier` / `format_weight`；
2. 全局重算：删除这两列，并在规则说明中写明「修改规则会重算全部历史积分」。

---

### M-3 🟠 登录缺少加固

**位置**：`server.js` `/api/login`（504 行口令比较）、`issueSession` / `setCookie`（115–120、174–178 行）。

**描述**：
- 口令比较 `password !== PASSWORD` 非常量时间，存在理论上的计时侧信道；
- `/api/login` 无失败次数限制，可被暴力枚举；
- `issueSession` 未传 `secure`，即使经 HTTPS 反代，会话 Cookie 也未带 `Secure` 标记。

**影响**：内网使用风险低；一旦「轻量公网」部署，单口令 + 无限流的组合较弱。

**建议**：登录失败按 IP 限流（如每分钟 N 次）；口令用常量时间比较（`crypto.timingSafeEqual`）；反代为 TLS 时给会话 Cookie 加 `Secure`。

---

### M-4 🟠 缺少操作审计日志

**位置**：全局。

**描述**：写操作均由单一共享口令授权，系统不记录「谁、何时、做了什么」。删除比赛为不可逆操作，目前无任何留痕，也无法区分操作人。

**影响**：误删 / 争议无法追溯。

**建议**：新增 `audit_log` 表（动作、目标、时间、来源 IP、可选操作人标签），至少对新增 / 删除比赛与球员维护落痕。

---

### L-1 🟡 每次保存球员 id 全部重排

**位置**：`server.js` `rewriteState`（375 行 `DELETE FROM players` + 381–384 行自增重灌）。

**描述**：全量重写会删表重灌，球员 id 由自增重新分配，同批次内靠 `oldToNew` 重映射，但 id 在每次保存后都会变化。

**影响**：当前自包含前端每次全量刷新可容忍；第二阶段做球员详情页 / 固定链接 / 外部引用时，id 不稳定将成为障碍。

**建议**：随 H-1 改为增量写入后，id 自然稳定。

---

### L-2 🟡 监听地址与日志不符

**位置**：`server.js` 547–548 行。

**描述**：`http.createServer(route).listen(PORT, ...)` 未指定 host，实际绑定 `0.0.0.0`（所有网卡），但日志打印 `http://127.0.0.1:3000`，易让人误判为仅本机可达。

**影响**：可能误导安全判断。容器 / 公网部署本就需对外，行为本身无误，问题在日志。

**建议**：将日志改为真实监听地址，或显式声明对外监听。

---

### L-3 🟡 核心算分引擎无单元测试

**位置**：全局。

**描述**：积分为路径依赖逻辑，又存在「现算系数」等易回归点，仓库未见对应测试。

**影响**：调参或重构时无回归保护。

**建议**：补充用例锁定不变量：零和恒等、各赛制各档系数、删除后重排一致、单打 1v1 与双打 2v2 计算正确。

---

### L-4 🟡 内存会话不清理、重启即全体登出

**位置**：`server.js` `sessions`（Map）。

**描述**：会话存于内存 Map，过期项仅在被访问时删除；进程重启后全部会话失效。

**影响**：轻微内存堆积；重启需重新登录（可接受）。

**建议**：作为已知行为记录即可；如需持久会话，可改存数据库或加定期清理。

---

### L-5 🟡 内联实时数据的页面被设为可缓存

**位置**：`server.js` `sendHtml`（167 行 `Cache-Control: public, max-age=30, s-maxage=60`）。

**描述**：首页内联了首屏排行榜数据，却设置了 `public` / `s-maxage=60`，经 CDN / 共享缓存时排行榜最多可能陈旧 60 秒。

**影响**：数据本就公开，风险低；仅影响实时性。

**建议**：若上 CDN，建议页面改为 `no-store` 或仅内联摘要、明细走接口拉取。

## 5. 整改优先级建议

| 优先级 | 项目 | 理由 |
|---|---|---|
| P0（尽快） | H-1 并发写入、M-1 比分校验 | 前者威胁数据安全，后者威胁数据质量，成本均低 |
| P1（公网前必做） | M-3 登录加固、M-4 审计日志 | 公网暴露 + 不可逆删除的前提条件 |
| P2（随二阶段） | M-2 快照一致性、L-1 id 稳定、L-3 单元测试 | 配合详情页 / 历史页与长期维护 |
| P3（按需） | L-2 / L-4 / L-5 | 健壮性与规范性改进 |
